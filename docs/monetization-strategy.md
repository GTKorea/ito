## 제품 배포 형태

ito는 두 가지 형태로 제공됩니다.

```
┌──────────────────────────────────────────────────────┐
│                                                      │
│  ito Cloud (itothread.com)                           │
│  ━━━━━━━━━━━━━━━━━━━━━━━━                            │
│  ito가 직접 운영하는 SaaS.                            │
│  가입하면 바로 사용 가능.                              │
│  모든 워크스페이스가 같은 인프라 위에 있으므로          │
│  SharedSpace(크로스 워크스페이스 협업)가 가능.          │
│                                                      │
│  예: Slack, Linear, Notion처럼 가입해서 쓰는 형태      │
│                                                      │
├──────────────────────────────────────────────────────┤
│                                                      │
│  ito Community Edition (셀프호스팅)                    │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━                    │
│  오픈소스(AGPL-3.0)로 공개된 코드를                    │
│  사용자가 자기 서버에 직접 설치해서 운영.               │
│  Docker Compose로 배포 가능.                           │
│                                                      │
│  예: GitLab CE, Mattermost처럼 직접 서버에 설치        │
│                                                      │
└──────────────────────────────────────────────────────┘
```

**핵심 차이**: 셀프호스팅 인스턴스는 각각 독립된 DB를 가지므로,
서로 다른 인스턴스 간 데이터 연동이 불가능합니다.

## SharedSpace가 만드는 자연스러운 업셀

### SharedSpace란?

SharedSpace는 **서로 다른 워크스페이스(= 다른 회사/팀)** 가 하나의 공간에서
태스크를 공유하고 스레드 체인을 연결할 수 있는 기능입니다.

```
┌─────────────┐              ┌─────────────┐
│  A사 워크스페이스 │◄─── SharedSpace ───►│  B사 워크스페이스 │
│  (디자인팀)   │    공유 태스크 공간    │  (개발팀)    │
└─────────────┘              └─────────────┘
                     │
                     ▼
              A사 디자이너가 B사 개발자에게
              스레드 체인으로 작업 연결 가능
```

### 왜 셀프호스팅에서는 SharedSpace가 안 되는가

```
셀프호스팅 인스턴스 A (A사 서버)     셀프호스팅 인스턴스 B (B사 서버)
┌─────────────────────┐          ┌─────────────────────┐
│  DB: PostgreSQL A    │          │  DB: PostgreSQL B    │
│  Users: A사 직원들    │    ✗     │  Users: B사 직원들    │
│  Tasks: A사 태스크    │◄──연결불가──►│  Tasks: B사 태스크    │
└─────────────────────┘          └─────────────────────┘

→ 각자 다른 서버, 다른 DB이므로 크로스 워크스페이스 협업 불가
→ A사와 B사가 SharedSpace를 쓰려면 같은 인스턴스에 있어야 함
```

```
ito Cloud (itothread.com)
┌──────────────────────────────────────────┐
│  DB: 단일 PostgreSQL                      │
│                                          │
│  A사 워크스페이스 ◄─── SharedSpace ───► B사 워크스페이스 │
│       │                                  │
│       └──── C사, D사, ... 추가 가능 ──────┘ │
│                                          │
│  → 같은 인스턴스이므로 크로스 협업 가능     │
└──────────────────────────────────────────┘
```

### 네트워크 효과 → 자연스러운 바이럴

```
1. A사가 ito Cloud 사용 시작
          │
2. A사가 외부 파트너 B사와 협업 필요
          │
3. SharedSpace 생성 → B사를 초대
          │
4. B사도 ito Cloud 가입 (무료든 유료든)
          │
5. B사가 자체 워크스페이스에서도 ito 사용 시작
          │
6. B사가 C사와 SharedSpace 생성 → C사 가입 → ...
          │
      네트워크 효과로 사용자 자연 증가
```

이것은 Slack이 성장한 방식과 동일합니다:
- 한 팀이 Slack을 쓰기 시작하면 → 협업하는 외부 팀도 Slack Connect로 연결
- 연결된 팀이 자기 팀에서도 Slack 도입 → 바이럴 성장

## 기능별 Free / Pro 구분

### 원칙

1. **핵심 기능은 무료** — 스레드 체인, snap-back은 ito의 정체성. 제한하면 안 됨
2. **네트워크 효과 기능은 클라우드 전용** — SharedSpace가 업셀 드라이버
3. **관리/보안 기능은 Pro** — 엔터프라이즈가 필요로 하는 것
4. **셀프호스팅 CE에도 충분한 가치** — 사내 팀 단위 협업은 완전 무료

### 기능 매트릭스

| 기능 | Community (셀프호스팅) | Cloud Free | Cloud Pro |
|------|:---------------------:|:----------:|:---------:|
| **스레드 체인 + snap-back** | O | O | O |
| **블로커 관리** | O | O | O |
| **Public/Private 그룹** | O | O | O |
| **태스크별 채팅 + 파일** | O | O | O |
| **Slack 통합 (기본)** | O | O | O |
| **데스크톱 앱 (Tauri)** | O | O | O |
| **9개 언어 지원** | O | O | O |
| **워크스페이스 멤버 수** | 무제한 | 10명 | 무제한 |
| **파일 스토리지** | 자체 서버 | 1GB | 50GB |
| **SharedSpace (크로스 워크스페이스)** | **X** | **O (1개)** | **O (무제한)** |
| **팀(Team) 관리** | O | O | O |
| **SSO / SAML** | X | X | O |
| **감사 로그 (Audit Log)** | X | X | O |
| **고급 분석 대시보드** | X | X | O |
| **우선 기술 지원** | X | X | O |
| **자동 백업 + 복원** | X | X | O |
| **Google Calendar 연동** | O | O | O |
| **커스텀 Webhook** | X | X | O |
| **API Rate Limit** | — | 기본 | 10x |

### Cloud Free에 SharedSpace 1개를 주는 이유

- 완전히 막으면 네트워크 효과 발생 안 됨
- 1개를 무료로 주면 → "이거 좋네, 다른 팀하고도 하고 싶다" → Pro 전환
- Slack의 "Slack Connect 1채널 무료"와 같은 전략

## 셀프호스팅 사용자 전환 시나리오

### 시나리오 1: 스타트업 팀 (5명)

```
1. GitHub에서 ito CE 발견 → Docker Compose로 셀프호스팅
2. 사내 태스크 관리에 사용 (무료로 충분)
3. 외주 디자인 에이전시와 협업 필요 발생
4. SharedSpace가 필요 → ito Cloud Free 가입 (10명, SharedSpace 1개)
5. 팀이 15명으로 성장 → Cloud Team으로 업그레이드
```

### 시나리오 2: 중견기업 개발팀 (30명)

```
1. ito CE 셀프호스팅으로 팀 내부 사용
2. 다른 부서(마케팅팀)도 사용하고 싶어함
3. SSO 통합이 필요 → Cloud Enterprise 문의
4. 데이터 마이그레이션 지원 → ito Cloud 전환
```

### 시나리오 3: 에이전시 (클라이언트 다수)

```
1. 클라이언트 A가 ito Cloud 사용 중
2. SharedSpace로 에이전시를 초대
3. 에이전시가 ito Cloud 가입 (무료)
4. 클라이언트 B, C, D도 SharedSpace로 연결
5. SharedSpace 1개 제한 → Team 플랜 업그레이드
```

**공통 패턴**: 외부 협업 필요성 → SharedSpace → ito Cloud → 유료 전환

| **Plausible** | O | 클라우드 편의성 | $2M+ ARR |
| **ito (계획)** | O (CE) | **SharedSpace** + 엔터프라이즈 보안 | — |

### ito의 강점

ito의 업셀 드라이버인 SharedSpace는 **네트워크 효과**를 만듭니다.
이것은 단순히 "기능을 더 준다"가 아니라 "더 많은 사람이 쓸수록 가치가 올라간다"입니다.

- GitLab의 CI/CD는 기능 업셀 (더 많은 기능 = 더 비싼 플랜)
- Slack Connect는 네트워크 업셀 (더 많은 연결 = 더 비싼 플랜)
- **ito의 SharedSpace는 Slack Connect와 같은 네트워크 업셀**

이 모델이 장기적으로 더 강력합니다. 기능 업셀은 경쟁자가 기능을 따라하면 끝나지만,
네트워크 업셀은 사용자가 많아질수록 경쟁자가 따라잡기 어렵습니다.

## 요약

```
오픈소스 (AGPL-3.0)
  → 개발자 커뮤니티 확보 + GitHub Stars 마케팅
  → 셀프호스팅으로 사내 사용 (무료)
  → 외부 협업 필요 시 ito Cloud로 자연 전환

SharedSpace = 네트워크 효과 엔진
  → 한 회사가 쓰면 → 협업 회사도 가입
  → 셀프호스팅에서는 불가능 → 클라우드 전용
  → Slack Connect와 동일한 바이럴 성장 구조

수익화 = Open Core + 네트워크 업셀
  → Community Edition: 사내 협업 무료
  → Cloud Free: SharedSpace 1개 체험
  → Cloud Team ($8/user/mo): 무제한 SharedSpace
  → Cloud Enterprise: SSO, 감사 로그, 전담 지원
```
